# Версионирование протокола AFB ↔ BF

Здесь два независимых уровня версий. Не путать.

## 1. Версия провода — `protocol`

Значение поля конверта `protocol`, сейчас `afb.execution.v1`. Это **контракт
совместимости на проводе**. Меняется (`v1` → `v2`) только при несовместимом
изменении формата конверта или семантики, при котором старый и новый получатель
не могут договориться. Обе стороны на сессии обязаны использовать одну версию
провода (проверяется при handshake и в `validate_envelope_fields`).

## 2. Версия пакета/спеки — semver git-тег

Тег вида `vX.Y.Z` (например `v1.0.0`), он же `__version__` пакета и
`info.version` в `spec/asyncapi.yaml`. AFB и BF пинят зависимость по тегу:

```
afb-bf-protocol @ git+https://github.com/Rolo837/AFB-BF-protocol.git@v1.0.0
```

**⚠️ MAJOR запрещён по умолчанию.** Поднимать MAJOR-версию (`v1.x → v2.0.0`)
запрещено без явной специальной команды пользователя — независимо от того,
насколько изменение формально несовместимо. Если задача требует несовместимого
изменения (удаление/переименование типа или поля, новое обязательное поле,
смена семантики), работа над этим изменением не выполняется без отдельного
разрешения пользователя на MAJOR; вместо этого нужно искать обратносовместимый
путь (новое опциональное поле вместо переименования обязательного, новый тип
вместо изменения семантики старого и т.п.).

Правила инкремента:

| Уровень | Когда | Действие потребителя |
|---------|-------|----------------------|
| **PATCH** (`v1.x.y → v1.x.(y+1)`) | Доки, исправления схем/кода без изменения формата на проводе; **любые расширения протокола, не пересекающие канал AFB↔BF** — в т.ч. схемы/конвенции, специфичные для связки фронтенд↔бэкенд AFB (например `notification.*.vN.json`: они не входят в `spec/asyncapi.yaml`, не подписываются и никогда не пересекают канал AFB↔BF) | безопасно обновиться |
| **MINOR** (`v1.x.y → v1.(x+1).0`) | Обратносовместимое изменение протокола **AFB↔BF**: новый тип сообщения, новое **опциональное** поле, новый enum-вариант для исходящих — не ломающее взаимодействие с существующими BF | безопасно обновиться; новые возможности опциональны |
| **MAJOR** (`v1.x → v2.0.0`) | Несовместимое изменение: удаление/переименование типа или поля, новое обязательное поле, смена семантики; обычно вместе с новым `protocol` `vN`. **Запрещено без специальной команды пользователя** (см. выше) | требуется согласованное обновление обеих сторон |

**Связь уровней:** смена версии провода (`afb.execution.vN`) всегда влечёт MAJOR
пакета — то есть возможна только вместе с явно разрешённым MAJOR. MAJOR пакета
не обязательно меняет версию провода (например, удаление давно депрекейтнутого
типа в рамках того же `v1`), но всё равно требует разрешения.

## 3. Совместимость сторон

- AFB и BF должны быть на одной **MAJOR** версии пакета.
- В пределах одной MAJOR более новая сторона обязана понимать сообщения от более
  старой (forward/backward в рамках MINOR/PATCH).
- Источник истины статусов сделки — AFB; источник истины о состоянии у брокера —
  BF (см. `docs/` и `spec/`).

## 4. Депрекейт типов сообщений

1. Пометить сообщение в `spec/asyncapi.yaml` тегом `deprecated` и описать замену
   в `summary`/`description`. (Прецедент миграции имён: `bf.*`/`execution_deal.*`
   → `session.*`/`deal.*`.)
2. Релиз MINOR: тип ещё принимается, но помечен; обе стороны переводятся на
   замену.
3. Удаление типа — только в следующем **MAJOR** (запрещён по умолчанию, см. §2 —
   нужна отдельная команда пользователя), после того как обе стороны перестали
   его слать (подтверждается журналами `trading_events`).

## 5. Процедура релиза

1. Изменить `spec/asyncapi.yaml` и/или `spec/schemas/*` (канон).
2. `afb-bf-protocol-generate` — перегенерировать `taxonomy.py`, `docs/` и
   `ts/src/taxonomy.ts` + `ts/src/index.ts` (и `ts/src/models.ts`, если есть
   Node-генерация моделей — см. `ts/tools/generate-models.mjs`).
3. `python -m afb_bf_protocol.tools.make_fixtures` — обновить примеры при
   изменении формата.
4. `pytest` (схемы, подписи, эквивалентность) + `npx @asyncapi/cli validate
   spec/asyncapi.yaml` + `npm run typecheck` — зелёные.
5. Поднять версию в `pyproject.toml`, `version.py`, `info.version` спеки и
   `package.json`.
6. Тег `vX.Y.Z`, push. Потребители пинят новый тег (включая
   `AFB/frontend/package.json`, если он уже переведён на пакет).
